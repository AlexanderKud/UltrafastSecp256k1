## Why this plan exists (TASK-001 / TASK-002 / BENCH-101..103)

The 2026-05-22 multi-pass review surfaced three coupled bench-integrity
findings that the documentation pipeline alone cannot close:

* **BENCH-101 (P0):** the file pointed at by
  `docs/canonical_numbers.json._canonical_bench_artifact` —
  `docs/bench_unified_2026-05-21_gcc14_x86-64_v2.json` — is missing the
  `metadata.generated_by: "bench_unified --json"` field that the policy
  block in `canonical_numbers.json` claims it has. The artifact was
  either hand-edited or generated by an older binary; neither is
  reproducible.
* **BENCH-102 (P0):** the BENCH-001 fix in
  `src/cpu/bench/bench_unified.cpp:2534` (label `inv (fast/var-time)`
  replacing the previous duplicated `inv (CT)` rows with identical ns)
  is present in source but NOT reflected in the v2 JSON — the JSON
  still emits both `"inv (CT)"` and `"inv (var-time)"` with ns values
  within 5% of each other. Source-vs-data divergence.
* **BENCH-103 (P0):** `docs/BITCOIN_CORE_BACKEND_EVIDENCE.md` and
  `docs/BENCHMARKS.md` still cite `1.27×` ECDSA / `1.13×` Schnorr
  ratios while the v2 JSON's own arithmetic is `1.23× / 1.11×`.
  `canonical_numbers.json` records `1.23 / 1.11`, but the prose docs
  diverge — `check_bench_doc_consistency.py` only validates
  `canonical_numbers.json ↔ bench JSON`, not the prose tables.

All three require a fresh `bench_unified --json` run on a quiet host
and a propagation pass through the existing sync scripts.

## What to run (single owner-side session)

Pin the host first — the values below are validated only on this
configuration class (Intel Skylake or newer, single-core pinned,
turbo locked off, governor=performance):

```bash
# Pin to performance governor.
sudo cpupower frequency-set -g performance

# Disable turbo so the measurement is reproducible across runs.
echo 1 | sudo tee /sys/devices/system/cpu/intel_pstate/no_turbo

# Build fresh — never reuse an older bench_unified binary.
cmake -S . -B out/bench -G Ninja \
    -DCMAKE_BUILD_TYPE=Release \
    -DCMAKE_CXX_COMPILER=g++-14 \
    -DCMAKE_CXX_FLAGS="-O3 -march=native -DNDEBUG" \
    -DSECP256K1_USE_LTO=ON
cmake --build out/bench --target bench_unified -j"$(nproc)"

# Pin to core 0, max scheduling priority. The `nice -20` matters —
# without it OS jitter widens the IQR by ~3×.
TODAY=$(date +%Y-%m-%d)
ARTIFACT="docs/bench_unified_${TODAY}_gcc14_x86-64_v2.json"
taskset -c 0 nice -20 ./out/bench/cpu/bench_unified --json > "${ARTIFACT}"

# Sanity-check the artifact carries the required metadata.
jq -e '.metadata.generated_by == "bench_unified --json"' "${ARTIFACT}" \
    || { echo "FAIL: metadata.generated_by missing — regen aborted"; exit 1; }
jq -e '.metadata.date' "${ARTIFACT}"  # not null
```

## Propagation (auto-syncs from the canonical JSON)

The repo has three sync scripts that read `canonical_numbers.json` and
update the prose documents. Run them in order:

```bash
# 1. Update canonical_numbers.json's ratio fields from the new artifact.
#    (Hand-edit the ratio fields if `sync_canonical_numbers.py` cannot
#     auto-derive them — the script asks for confirmation on > 5% deltas.)
python3 ci/sync_canonical_numbers.py "${ARTIFACT}"

# 2. Propagate the ratios to every doc that pulls from canonical_numbers.json.
python3 ci/sync_docs_from_canonical.py

# 3. Verify nothing drifted.
python3 ci/check_bench_doc_consistency.py
```

## Why this file isn't a script

The actual benchmark must be supervised — the run can drift if any of
the following are wrong, and silent acceptance of bad data is exactly
the failure mode BENCH-101/102/103 caught:

* governor != performance → mean shifts ~5-15%
* turbo unlocked → IQR widens ~10× and the median is meaningless
* core not pinned → OS migrates between cores, cache-state varies
* concurrent workload on the host → measurements jitter
* `nice` not applied → median picks up scheduling latency tails

A wrapper script that silently runs these would let a future engineer
generate a "fresh" JSON on an unpinned host and ship it as canonical.
That is the exact category of regression this file is meant to prevent.

## Acceptance criteria for the regenerated artifact

A regen is COMPLETE only when ALL of the following hold:

- [ ] `jq '.metadata.generated_by'` returns `"bench_unified --json"`
- [ ] `jq '.metadata.date'` is non-null and within 24h of the file's mtime
- [ ] No row named `"inv (CT)"` has a ns value within 5% of any
      `"inv (var-time)"` row (BENCH-102 closed)
- [ ] `ci/check_bench_doc_consistency.py` passes
- [ ] `canonical_numbers.json` `_canonical_bench_artifact` points at
      the new file
- [ ] The previous `_canonical_bench_artifact` file is either deleted
      OR moved under `docs/archive/` (BENCH-103 / REL-102 closed —
      no two "canonical" files for the same date+config)
- [ ] `docs/BITCOIN_CORE_BACKEND_EVIDENCE.md` and `docs/BENCHMARKS.md`
      ratios match the JSON within 1% (after the sync scripts run)

Until all six boxes tick, the regen is partial and the BENCH-101/102/103
findings stay open.

## Bench-only build profile (TASK-001 supporting infrastructure)

A dedicated CMake preset (`cmake --preset bench`) would help avoid
flag drift across regenerations. The `cuda-release` and `cpu-release`
presets already encode safe defaults; a `bench` preset that inherits
`cpu-release` and forces `SECP256K1_USE_LTO=ON` +
`SECP256K1_BENCH_LOCK_TURBO=ON` would centralise the bench
configuration. Not yet added — owner discretion whether to land a
preset entry or keep the explicit-flag invocation.
